# DAUM AI WORKFLOW 작업 원칙

## 목적

이 문서는 다움 목공 자재 산출 프로그램에서 AI 작업자가 따라야 할 기본 작업 기준을 정의한다.
모든 작업은 기존 앱 동작을 보존하고, 요청된 범위 안에서만 수행한다.

## 핵심 원칙

1. 요청 범위를 벗어나지 않는다.
   - 사용자가 지정한 파일, 기능, 작업 범위 안에서만 변경한다.
   - 관련 없는 리팩터링, 구조 변경, 파일 이동, 이름 변경을 하지 않는다.

2. 기존 기능을 보존한다.
   - 계산식, UI, 저장 로직, Supabase 연동은 명시 요청 없이는 수정하지 않는다.
   - 기존 산출 결과가 달라질 수 있는 변경은 반드시 별도 지시가 있을 때만 수행한다.

3. 문서 작업과 코드 작업을 분리한다.
   - 문서 파일 추가 또는 수정 요청에서는 앱 코드, 계산식, UI, 저장 로직을 변경하지 않는다.
   - 문서-only 작업 후에는 실제 변경 파일이 문서에 한정되는지 확인한다.

4. 변경 전 상태를 확인한다.
   - 작업 전 `git status`로 현재 변경 상태를 확인한다.
   - 기존 사용자 변경이 있으면 덮어쓰지 않는다.

5. 최소 변경을 원칙으로 한다.
   - 문제 해결 또는 요청 이행에 필요한 최소 파일만 수정한다.
   - 새 파일은 요청되었거나 필요한 경우에만 추가한다.

6. 검증을 생략하지 않는다.
   - 코드 변경이 아니어도 사용자가 요청하면 `npm run build`를 실행한다.
   - 빌드 실패 시 원인과 변경 파일과의 관련성을 구분해 보고한다.

7. 보고는 한국어로 명확하게 작성한다.
   - 원인 또는 분석, 변경 파일, 수정 내용, 검증 결과, 빌드 결과, 커밋 및 푸시 결과를 구분한다.
   - 추측이 아니라 실제 확인한 결과만 보고한다.

## 금지 사항

- 명시 요청 없는 `app/page.tsx` 수정 금지
- 명시 요청 없는 계산식 수정 금지
- 명시 요청 없는 UI 수정 금지
- 명시 요청 없는 Supabase 저장/조회 로직 수정 금지
- 기존 기능을 바꾸는 리팩터링 금지
- 작업 범위 밖 파일 정리 또는 포맷팅 금지

## 작업 절차

1. 요청 범위와 금지 사항을 확인한다.
2. 현재 git 상태를 확인한다.
3. 필요한 파일만 생성 또는 수정한다.
4. 변경 파일 목록을 확인한다.
5. 요청된 검증 명령을 실행한다.
6. 검증 통과 후 요청이 있을 때만 `git add`, `git commit`, `git push`를 수행한다.
7. 최종 보고는 사용자가 요청한 형식을 따른다.
