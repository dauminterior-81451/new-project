# 다움 프로젝트 공통 용어와 결정 기록 원칙

## 목적

이 문서는 다움 목공 자재 산출 프로그램에서 사용하는 공통 용어와 의사결정 기록 기준을 정리한다.
AI 작업자와 개발자는 기능 변경 전 이 문서를 확인하고, 용어와 기록 기준을 일관되게 유지한다.

## 공통 용어

### 프로젝트

- 다움 프로젝트: 다움 목공 자재 산출 프로그램 전체를 의미한다.
- 앱: 사용자가 목공 자재 산출, 저장, 조회, 출력에 사용하는 웹 애플리케이션을 의미한다.
- 작업 기준 문서: AI 또는 개발자가 작업 전에 확인해야 하는 AGENTS.md, CONTEXT.md 등의 문서를 의미한다.

### 목공 자재 산출

- 산출: 입력값을 기준으로 목공 자재 수량, 규격, 금액 등을 계산하는 행위를 의미한다.
- 계산식: 산출 결과를 만드는 모든 수식, 조건, 보정 로직을 의미한다.
- 자재: 산출 대상이 되는 목공 자재 항목을 의미한다.
- 규격: 자재의 길이, 폭, 두께, 단위 등 산출에 필요한 기준 정보를 의미한다.
- 수량: 계산식 또는 사용자 입력에 의해 결정되는 자재 개수를 의미한다.
- 결과값: 계산식 실행 후 화면, 저장 데이터, 출력 데이터에 반영되는 값을 의미한다.

### UI와 저장

- UI: 사용자가 보는 화면 구성, 입력 폼, 버튼, 결과 표시 영역, 안내 문구를 의미한다.
- 저장 로직: Supabase 또는 기타 저장소에 데이터를 생성, 조회, 수정, 삭제하는 모든 흐름을 의미한다.
- Supabase: 프로젝트의 데이터 저장 및 조회에 사용되는 백엔드 서비스를 의미한다.
- 기존 기능: 현재 사용자에게 제공 중인 화면, 계산, 저장, 조회, 출력 기능을 의미한다.

### 문서 작업

- 문서 파일 추가: 코드, 계산식, UI, 저장 로직을 건드리지 않고 문서 파일만 생성하거나 수정하는 작업을 의미한다.
- 작업 기록: 변경 이유, 변경 파일, 검증 결과, 빌드 결과, 커밋 및 푸시 결과를 남기는 보고를 의미한다.

## 결정 기록 원칙

1. 결정은 기능보다 먼저 기록한다.
   - 계산식, UI, 저장 로직에 영향을 주는 결정은 구현 전에 문서 또는 이슈에 먼저 남긴다.

2. 결정의 배경을 함께 기록한다.
   - 무엇을 결정했는지만 쓰지 않고, 왜 그 결정이 필요했는지와 어떤 제약이 있었는지 함께 적는다.

3. 계산식 변경은 별도 결정으로 취급한다.
   - 산출 결과에 영향을 주는 계산식 변경은 사소한 수정으로 취급하지 않는다.
   - 변경 전후 기준, 영향 범위, 검증 방법을 반드시 기록한다.

4. UI 변경은 사용자 영향 범위를 기록한다.
   - 화면 배치, 문구, 버튼, 입력 흐름 변경은 사용자가 체감하는 변경으로 보고 기록한다.

5. 저장 로직 변경은 데이터 영향 범위를 기록한다.
   - Supabase 테이블, 컬럼, 저장 형식, 조회 조건, 삭제 조건 변경은 데이터 영향 범위를 명확히 남긴다.

6. 문서-only 작업은 코드 변경과 분리한다.
   - 문서 파일만 추가하거나 수정하는 작업에서는 앱 코드, 계산식, UI, 저장 로직을 함께 변경하지 않는다.

7. 검증 결과를 결론형으로 기록한다.
   - `npm run build 통과`, `문서 파일만 변경 확인`, `기존 앱 코드 변경 없음`처럼 재확인 가능한 문장으로 남긴다.

## 기록 형식 권장안

작업 보고는 다음 항목을 기본으로 한다.

- 원인 또는 분석
- 변경 파일
- 수정 내용
- 검증 결과
- npm run build 결과
- git commit
- push 결과
